# 문제 분해 (Problem Decomposition)

## 해결해야 할 문제
$ARGUMENTS

## 분석 요청

위 문제를 다음 단계에 따라 체계적으로 분해해주세요:

### 1단계: 문제 정의
- **문제의 핵심**: 이 문제가 실제로 해결하려는 것은 무엇인가?
- **입력**: 무엇이 주어지는가?
- **출력**: 무엇을 만들어야 하는가?
- **성공 기준**: 어떻게 되면 "완료"인가?
- **제약조건**: 지켜야 할 규칙이나 한계는?

### 2단계: 하위 작업 분해
문제를 독립적으로 구현 및 테스트 가능한 하위 작업들로 나눠주세요.

각 작업에 대해 다음을 명시:
- **작업 ID**: T1, T2, T3...
- **작업명**: 간결한 제목
- **목표**: 이 작업이 달성하는 것
- **선행 작업**: 의존하는 작업 ID (없으면 "없음")
- **예상 변경 범위**: 영향받는 파일/모듈
- **완료 기준**: 체크리스트 형태로

### 3단계: 실행 순서 도출
- 의존성 그래프를 기반으로 최적의 실행 순서 제시
- 병렬로 진행 가능한 작업 그룹 식별

### 4단계: 리스크 및 고려사항
- 기술적 리스크나 불확실성
- 추가 조사가 필요한 부분
- 의사결정이 필요한 지점

## 출력 형식

```
## 문제 요약
[한 문장 요약]

## 하위 작업 목록

### T1: [작업명]
- 목표: ...
- 선행 작업: 없음
- 변경 범위: ...
- 완료 기준:
  - [ ] ...
  - [ ] ...

### T2: [작업명]
- 목표: ...
- 선행 작업: T1
- 변경 범위: ...
- 완료 기준:
  - [ ] ...

## 실행 순서
1. T1 (독립)
2. T2, T3 (병렬 가능)
3. T4 (T2, T3 완료 후)

## 리스크 및 고려사항
- ...
```

## 추가 지침
- 각 작업은 1-2시간 내 완료 가능한 크기로 분해
- 작업명은 동사로 시작 (예: "구현", "추가", "수정")
- 불확실한 부분은 명시적으로 표시
- 현재 코드베이스의 구조와 패턴을 고려하여 분해
